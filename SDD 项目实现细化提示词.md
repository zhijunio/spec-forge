## 一、项目背景

我要设计一个新的 SDD 项目。它不是单纯的代码生成器，而是一个**面向多 Agent 协作的软件开发工作流 / 项目模板 / 规则体系**。

这个项目的核心方向是：

- **skill-first，只保留必要入口约定**
- 自动识别并加载技能
- 支持子代理并行协作
- 默认保持**原子提交**；`worktree` 为可选能力，默认不启用
- 流程中必须包含 **TDD**
- 全流程覆盖：
  **discover → spec → plan → build → review → simplify → release**
- 尽量减少文档数量，但每份文档职责必须明确
- `agent.md` / `claude.md` 为同义文件，内容应保持一致
- 需要一个**轻量初始化机制**，用于建立项目上下文入口，而不是生成一堆重型模板
- 适合跨 agent 平台使用，而不是绑定单一产品

## 五、执行流程约束

推荐主流程：

1. init
2. discover
3. spec
4. plan
5. build
6. review
7. simplify
8. release

补充约束：

- **TDD 是硬规则**
- 一个任务尽量由一个 **fresh subagent** 执行
- review 建议分两层：
  - **spec compliance review**
  - **code quality review**
- 子代理应尽量减少上下文污染
- 正式文档尽量少，但入口要稳定、清晰、机器友好

### 阶段职责契约

### 阶段开头模板

每个阶段的介绍建议统一用同一套短开头，避免自由发挥导致职责漂移：

- `Overview`：一句话说明这个阶段做什么
- `When to Use`：什么时候进入这个阶段
- `Inputs`：该阶段依赖什么输入
- `Outputs`：该阶段必须产出什么
- `Process`：只列最少的执行步骤
- `Verification`：如何判断完成
- `Red Flags`：哪些行为说明越界或失败

原则：

- 开头保持短，不写长背景
- 每个阶段都先讲边界，再讲过程
- 介绍只服务执行，不服务叙事

### 阶段交接规则

技能之间的交接要尽量小、尽量硬、尽量可回退：

- `Handoff To`：下一阶段是谁
- `Handoff Payload`：只传下一阶段必需的最小信息
- `Exit Criteria`：当前阶段什么时候算完成
- `Rollback Rule`：交接失败时回到哪一阶段

原则：

- 不传整段上下文，尽量只传交接包
- 交接时必须带冻结范围
- 交接失败优先回退到上一阶段，不在当前阶段硬凑
- 每次交接都要明确下一跳，不留歧义

#### `init`
- Overview: 只做项目识别、上下文入口确认和 `Project.md` 补全。
- 只做项目识别、上下文入口确认、`Project.md` 补全
- 不做需求收敛
- 不做任务拆解
- 不做实现决策

#### `discover`
- Overview: 只做问题发现、范围澄清和约束整理。
- 只做问题发现、范围澄清、约束整理
- 不写实现计划
- 不写任务清单
- 不把需求冻结成最终结论
- 不输出 `feature.md`

#### `spec`
- Overview: 只做需求收敛和验收标准定义，并输出 `feature.md`。
- 只做需求收敛和验收标准定义
- 明确范围、边界、不做什么、验收标准、关键场景
- 不写实现步骤
- 不写文件级任务拆解
- 不写 `plan.md`

#### `plan`
- Overview: 只做任务拆解、执行顺序、验证方式和子代理切分，并输出 `plan.md`。
- 只做任务拆解、执行顺序、验证方式和子代理切分
- 不改需求结论
- 不直接改代码

#### `build`
- Overview: 只按 `plan` 实现代码和测试，并默认走 TDD。
- 只做按 plan 实现代码和测试
- 以 TDD 为默认方式推进
- 不改需求边界
- 不重新设计整体计划

#### `review`
- Overview: 只审查 `spec`、`code quality` 和 `TDD` 过程，并冻结可修改范围。
- 只做 spec / code quality / TDD 过程审查
- 范围默认按“本次任务变更集”确定；只有用户显式指定时才改为文件 / 模块 / 提交级范围
- 默认只审查本次任务变更集，不默认扫全仓库
- 范围优先级：用户显式指定 > 当前任务变更集 > 分支 diff > 仓库全量
- 比较基线优先使用当前分支的上游或任务指定基线，不默认写死 `main` / `master`
- 输出格式固定为：`作用范围`、`冻结范围`、`处理项`、`问题清单`、`通过/驳回决定`
- `code quality` 默认按五维评审：正确性、可读性、架构、安全性、性能
- 问题分级默认用 `Critical`、`Important`、`Suggestion`、`Info`
- 分级结果可配合 emoji 图标展示，保持醒目但不喧宾夺主
- 不引入新需求
- 不扩展新计划
- 不直接替代 build

#### `simplify`
- Overview: 只在 review 已确认范围内做去复杂化、重构收口和冗余清理。
- 只做去复杂化、重构收口、冗余清理
- 只在 `review` 明确允许的范围内做修改
- 默认按以下维度判断是否简化：复杂度、可读性、重复、耦合、可测试性；性能只在 review 已指出或有明显收益时处理
- `review` 负责指出可简化范围和冻结范围，`simplify` 只执行已确认项
- 输出格式固定为：`作用范围`、`冻结范围`、`处理项`、`重构提交`、`结果说明`
- 不扩大功能范围
- 不重新打开已冻结的需求
- 不新增独立阶段

#### `release`
- Overview: 只做交付收尾、版本确认和发布准备。
- 只做交付收尾、版本确认、发布准备
- 不再引入新需求
- 不再拆新任务
- 不再做主体重构

### 默认产物表

| 阶段 | 输入 | 输出 | 禁止项 |
| --- | --- | --- | --- |
| `init` | 仓库、现有文档、用户触发 | 项目识别结果、`Project.md` 补全建议 | 需求文档、计划文档 |
| `discover` | 用户想法、现有上下文、`Project.md` | 问题列表、范围边界、约束、待澄清项 | `feature.md`、`plan.md` |
| `spec` | `discover` 结论、`Project.md` | `feature.md`、验收标准 | `plan.md`、实现步骤 |
| `plan` | `feature.md`、`Project.md`、`discover/spec` 结论 | `plan.md`、任务拆解、执行顺序、验证方式、子代理切分 | 代码、测试实现 |
| `build` | `plan.md`、`feature.md`、`Project.md` | 代码、测试、提交 | 新需求、重写计划 |
| `review` | `feature.md`、`plan.md`、`build` 产物、测试结果 | 审查结论、问题清单、通过/驳回决定 | 新需求、实现代码 |
| `simplify` | `build` 产物、`review` 结论 | 简化后的代码、清理结果、重构提交 | 新需求、计划扩展 |
| `release` | `review` 通过结论、`simplify` 结果、提交历史 | 发布确认、版本标记、交付完成状态 | 新需求、新任务 |
