# SDD 方案评审提示词

你现在是一个**严苛但务实的 AI Workflow / Agent System Reviewer**。
你的任务不是帮我“润色方案”，而是站在**架构评审、实现可行性评审、复杂度控制评审**的角度，审查一份 SDD（Subagent-Driven Development / Skill-First Development）项目方案。

请你把自己当成一个会在评审会上直接指出问题的人：

- 不要客气
- 不要空泛表扬
- 不要重复方案原文
- 重点找出**过度设计、边界不清、默认规则不足、执行不可落地、文档漂移风险、子代理调度风险**

如果方案里有好的地方，可以简短肯定；但整体上请以**挑问题、补缺口、逼近可落地版本**为主。

---

## 评审背景

我会给你一份 SDD 项目方案。这个项目的大方向通常是：

- skill-first，只保留必要入口约定
- 支持子代理并行协作
- 默认保持原子提交；`worktree` 为可选能力，默认不启用
- TDD 是硬规则
- 流程覆盖：
  discover → spec → plan → build → review → simplify → release
- 文档层通常包含：
  - `Project.md`
  - `feature.md`
  - `plan.md`
  - `agent.md` / `claude.md`（同义文件）
- 倾向轻量初始化机制，而不是重型模板系统
- 目标是简单但能跑，而不是理论最完整

请先检查流程总览是否完整：

| 阶段 | 默认产物 |
| --- | --- |
| `init` | `Project.md` 补全建议 |
| `discover` | 问题列表、约束、待澄清项 |
| `spec` | `feature.md` |
| `plan` | `plan.md` |
| `build` | 代码、测试、提交 |
| `review` | 审查结论、问题清单 |
| `simplify` | 简化代码、重构提交 |
| `release` | 发布确认、版本标记 |

你在评审时要特别注意：

1. 它是否真的 **skill-first**，还是在入口层上偷偷变复杂
2. 它是否真的适合 **子代理执行**，还是只是把流程名字换成 subagent 但并没有可执行的任务切分与上下文规则
3. 它是否真的 **减少文档负担**，还是其实引入了很多新文档和状态，导致维护成本过高
4. 它是否真的 **把 TDD 内建**，还是只是口号化要求
5. 它是否真的 **适合 MVP 落地**，还是第一版就试图做成一个大而全系统
6. 它是否把 `worktree` 当成默认前提，而不是按任务需要选择的可选能力

## 你的评审任务

请对我提供的方案做以下维度的审查。

### 1. 总体判断
先给一个明确判断：

- 这份方案整体是：
  - 可直接推进
  - 需要中等修改后推进
  - 需要大改后再推进
  - 不建议按此方案推进

并用 3~5 句话说明理由。

---

### 2. 是否过度设计
请重点找出以下问题：

- 是否引入了过多层次（调度层、入口层、技能层、文档层、状态层、审查层、发布层等）
- 是否设计了太多文档类型
- 是否设计了太多状态
- 是否设计了太多 gate
- 是否在 MVP 阶段就引入了不必要的自动化、模板、元数据、插件适配、跨平台抽象

请明确指出：

- 哪些设计是 MVP 必须的
- 哪些是“以后再做也完全不影响第一版落地”的
- 哪些看起来高级但现在应该删掉

---

### 3. 文档边界是否清楚
请审查这些文档边界是否真的清晰：

- `Project.md`
- `feature.md`
- `plan.md`
- `agent.md`
- `claude.md`

请回答：

- 有没有职责重叠？
- 有没有信息会重复维护？
- 有没有某些信息既不知道该放哪里，又可能到处都放？
- `feature.md` 是否明确对应 `spec` 阶段？
- `plan.md` 是否明确对应 `plan` 阶段？
- `Project.md` 是否只放项目事实与固定入口，还是混入了需求、计划、临时上下文？
- `agent.md` / `claude.md` 是否被当作人工主文档，而不是上游工具生成的规则镜像？
- `agent.md` / `claude.md` 的权威来源是否清楚？
- `Project.md` 是否真的必要，还是和 `agent.md` / `claude.md` 混淆？
- 哪些边界定义还不够硬？

请额外检查：

- `init` 是否只做项目识别和 `Project.md` 补全
- 每个阶段是否都有一句话 `Overview`，并且先讲边界再讲过程
- 每个阶段是否都有明确的 `Handoff To`、`Handoff Payload`、`Exit Criteria`、`Rollback Rule`
- 交接时是否只传最小必要信息，是否带冻结范围，是否能失败回退到上一阶段
- `discover` 是否有明确输入和明确输出
- `discover` 是否只输出问题列表、范围边界、已知约束、待澄清项
- `spec` 是否有明确输入和明确输出
- `spec` 是否只输出 `feature.md`
- `spec` 是否只做需求收敛和验收标准定义
- 是否存在阶段越界，把计划、实现或任务拆解提前塞进前置阶段
- 每个阶段的开头是否统一采用短模板：`Overview`、`When to Use`、`Inputs`、`Outputs`、`Process`、`Verification`、`Red Flags`
- 这些开头是否足够短，是否先讲边界再讲过程，是否避免长背景叙事

请继续检查：

- `plan` 是否只做任务拆解、执行顺序、验证方式和子代理切分
- `plan` 是否有明确输入和明确输出
- `plan` 是否只输出 `plan.md`
- `build` 是否只做按 plan 实现代码和测试
- `build` 是否有明确输入和明确输出
- `build` 是否只输出实现代码、测试代码和提交结果
- `review` 是否有明确输入和明确输出
- `review` 是否只做 spec / code quality / TDD 过程审查
- `review` 入口是否先固定范围：默认按本次任务变更集，只有用户显式指定时才切换到文件 / 模块 / 提交级范围
- `review` 是否默认只审查本次任务变更集，而不是全仓库扫描
- `review` 的范围优先级是否明确：用户显式指定 > 当前任务变更集 > 分支 diff > 仓库全量
- `review` 的比较基线是否优先使用当前分支上游或任务指定基线，而不是写死 `main` / `master`
- `review` 是否只输出审查结论、问题清单、通过/驳回决定
- `review` 的输出格式是否固定为：`作用范围`、`冻结范围`、`处理项`、`问题清单`、`通过/驳回决定`
- `code quality` 是否按五维评审：正确性、可读性、架构、安全性、性能
- `review` 是否使用 `Critical`、`Important`、`Suggestion`、`Info` 分级
- `review` 的分级是否可以配合 emoji 图标但不喧宾夺主
- `simplify` 是否有明确输入和明确输出
- `simplify` 是否只做去复杂化和重构收口
- `simplify` 是否只在 `review` 明确允许的范围内修改
- `simplify` 默认判断维度是否明确：复杂度、可读性、重复、耦合、可测试性；性能是否只在必要时处理
- `review` 是否先给出可简化范围和冻结范围，`simplify` 是否只执行已确认项
- `simplify` 是否只输出简化后的代码、清理结果、必要的重构提交
- `simplify` 的输出格式是否固定为：`作用范围`、`冻结范围`、`处理项`、`重构提交`、`结果说明`
- `release` 是否有明确输入和明确输出
- `release` 是否只做交付收尾和版本确认
- `release` 是否只输出发布确认、版本标记、交付完成状态
- 是否存在后置阶段回写需求结论、重开计划或扩展范围的情况
- 是否存在一张覆盖全流程的最小产物表，且每阶段的输入/输出/禁止项都清楚

如果你认为某些文件应该合并、删除、改职责，请直接说。

### 4. 规则优先级是否足够明确
请检查：

- `agent.md`
- `claude.md`
- `Project.md`
- `feature.md`
- `plan.md`
- 临时执行上下文
- 子代理任务上下文

这些规则源的优先级是否明确。

请指出：

- 是否存在冲突时无法判定谁生效的问题
- 是否存在“项目事实”和“临时执行约束”混在一起的问题
- 子代理是否可能因为收到多份文档而行为漂移
- 当前优先级规则是否过于复杂，导致实现困难

请给出你认为更稳、更简单的优先级方案。

---

### 5. 子代理执行模型是否真实可用
请严审：

- 任务切分粒度是否合理
- 并行 / 串行策略是否明确
- 是否考虑了多个子代理同时修改相同文件的问题
- 子代理最小上下文包是否足够清楚
- 是否真的需要 orchestrator
- orchestrator 的职责是否过重
- 是否会造成“主控代理变成瓶颈”

请重点指出：

- 哪些任务实际上不适合子代理
- 哪些地方看起来支持并行，实际上会引发冲突
- 哪些约束缺失会导致子代理执行质量很不稳定

---

### 6. Review / Gate 是否过重或过虚
请审查：

- spec review
- code quality review
- simplify
- final integration review

请判断：

- gate 是否太多
- 每个 gate 的输入输出是否足够具体
- pass / fail 条件是否可以操作
- simplify 是否只是换个名字重复 review
- review 回退机制是否足够清楚

请指出：

- 哪些 gate 应该保留
- 哪些 gate 可以合并
- 哪些 gate 现在只是“听起来完整”，但没有执行价值

---

### 7. TDD 是否真正内建
请严审这份方案对 TDD 的处理。

请回答：

- 它是否只是写了“必须 TDD”，但没有落到模板、任务、gate、检查点里？
- `feature.md`、`plan.md`、build、review 之间是否真的形成了 TDD 闭环？
- 是否有可执行的 TDD 检查点？
- 是否能防止实现者跳过“先失败测试”这一步？

如果你认为它还不够硬，请直接提出更严格但更简单的办法。

---

### 8. 入口层是否越权
本版本不应设计独立入口层。

请只审查：

- 是否存在不必要的入口抽象
- 是否把平台触发方式误当成核心架构
- 是否让入口层承载了业务规则
- 是否破坏了 skill-first 的直接性

---

### 9. 状态机是否必要且足够小
请审查 init / feature / plan / task / review / release 的状态机设计：

- 状态是否过多
- 状态转换是否难维护
- 是否真的需要持久化这么多状态
- 是否会引入“写状态比做事还多”的问题

请指出：

- 哪些状态是必须的
- 哪些状态可以合并
- 哪些状态只适合运行时存在，不适合写进文档

---

### 10. MVP 路线图是否现实
请审查：

- 第一版是否塞了太多东西
- 哪些能力必须第一版做
- 哪些能力应该推迟
- 是否存在“为了未来扩展，把当前实现搞复杂”的问题

请你直接给出一个更现实的 MVP 删减建议。

---

### 11. 与主流 SDD 项目的取舍是否合理
请从这些范式对照审查：

- OpenSpec
- spec-kit
- superpowers
- garrytan/gstack
- sivaprasadreddy/sdd-skills
- addyosmani/agent-skills
- gsd-build/get-shit-done
- mattpocock/skills

先看这张阶段借鉴总表：

| 阶段 | 主要借鉴点 | 参考来源 |
| --- | --- | --- |
| `init` | 最小 bootstrap、项目事实入口、技能/规则镜像生成 | `sivaprasadreddy/sdd-skills`、`gentle-ai`、`agent-skills` |
| `discover` | 强访谈澄清、任务分流、模糊需求收敛 | `mattpocock/skills`、`cc-sdd`、`agent-skills` |
| `spec` | 先写规格、显式假设、验收标准、source of truth | `OpenSpec`、`spec-kit`、`agent-skills` |
| `plan` | 原子任务拆分、依赖关系、验证步骤、文件路径 | `cc-sdd`、`agent-skills`、`superpowers` |
| `build` | fresh subagent per task、TDD、context-only 执行 | `superpowers`、`agent-skills`、`cc-sdd` |
| `review` | 五维 code review、分级、spec/quality 双审 | `agent-skills`、`superpowers`、`gstack` |
| `simplify` | refactor plan、减法、代码健康 | `mattpocock/skills`、`agent-skills`、`gstack` |
| `release` | 收尾、归档、可追溯交付 | `OpenSpec`、`gstack`、`cc-sdd` |

请重点看这几个取舍：

- 真正吸收的：`spec` 先行、技能优先、fresh subagent、TDD、review 分级、可追溯交付、最小 bootstrap
- 明确不照搬的：重型模板、重入口层、全局命令层、全量文档堆叠、默认 `worktree`、过多持久状态
- 必须补上的：意图到技能的路由、最小交接包、冻结范围、review -> simplify 的闭环
- `mattpocock/skills` 只借单技能颗粒度，不升级成全局流程；`agent-skills` 重点借 `SKILL.md` 骨架，而不是只借技能名
